# MMO Server Test

C#과 .NET 기반으로 구축한 실시간 MMO 게임 서버입니다. 단순한 기능 구현을 넘어,    
**대규모 트래픽 처리를 위한 비동기 구조**와 **데이터 무결성**을 고려한 멀티스레드 설계를 목표로 개발했습니다.

## 핵심 기능 설계

본 프로젝트는 재사용 가능한 네트워크 코어(ServerCore)와 실제 게임 로직이 돌아가는 컨텐츠 서버(Server),   
그리고 서버와 통신하는 클라이언트(Client)로 역할을 명확히 분리하여 설계하였습니다.

### 1. Server Core (네트워크 엔진부)
네트워크 통신의 기반이 되는 엔진으로, 특정 게임에 종속되지 않고 범용적으로 사용할 수 있도록 설계했습니다.   
가비지 컬렉터(GC)의 개입을 최소화하여 성능 저하(Spike)를 방지하는 데 집중했습니다.  

* **Listener**
    * 클라이언트의 연결 요청을 비동기적으로 처리
    * SocketAsyncEventArgs를 풀링하여 매번 객체를 생성/삭제하지 않고 재사용함 -> 메모리 할당 오버헤드 문제 개선
      
* **Session.cs & PacketSession.cs**
    * 수신된 바이트 배열에서 패킷 헤더(사이즈, ID)를 읽고, 하나의 패킷이 조립되었을 때만 컨텐츠 단으로 올려보내도록 처리
      
* **RecvBuffer.cs**
    * 데이터를 받을 때마다 버퍼를 새로 할당하지 않고, 메모리를 미리 할당해 둔 뒤 사용하는 방식을 채택

### 2. Server (MMO 컨텐츠 로직부)
Server Core 위에서 돌아가는 실제 게임 로직입니다.  
다수의 유저가 같은 공간에서 상호작용할 때 발생하는 **멀티스레드 동기화 문제**를 해결하는 데 초점을 맞췄습니다.

* **GameRoom.cs**
    * 유저들이 모이는 하나의 공간(Room)을 관리, 입장/퇴장, 이동, 스킬 사용 등의 로직이 이곳에서 처리
    * 멀티스레드 환경에서 몬스터, 플레이어 등에 동시 접근 시 발생하는 데드락이나 데이터 오염을 막기 위해 **JobQueue**를 도입
      
* **JobQueue.cs & JobTimer.cs**
    * 모든 로직을 즉시 실행(Lock)하지 않고, 실행할 함수(Action)를 Queue에 넣은 뒤 단일 스레드가 순차적으로 뽑아서 실행
    * JobTimer를 통해 지연 실행 로직을 추가하였을 때 서버의 프레임을 안정적으로 유지 할 수 있도록 사전에 보완
      
* **ClientSession.cs & PacketHandler.cs**
    * 네트워크 레이어에서 조립된 패킷이 구글 Protobuf를 거쳐 PacketHandler로 들어옴  
    * 패킷 ID에 따라 알맞은 로직으로 분기해주며, ClientSession은 현재 연결된 유저의 상태 정보와 네트워크 소켓을 매핑해주는 역할

### 3. Client (유니티 서버 연동부)
유니티 엔진의 **단일 스레드(Main Thread) 제약**을 고려하여 서버와 안전하게 데이터를 교환하도록 구성하였습니다.

* **ServerSession.cs (클라이언트 측 통신)**
    * 서버의 Session과 반대되는 개념으로, 유니티 클라이언트가 서버로 패킷을 보내고 받는 창구 역할
      
* **PacketQueue.cs**
    * 서버로부터 데이터를 수신하는 스레드(Background)와 게임 오브젝트를 조작하는 유니티 메인 스레드
    * 수신 스레드가 패킷을 바로 처리하지 않고 이 `PacketQueue`에 안전하게(Lock) 넣어두는 버퍼 역할
      
* **NetworkManager.cs**
    * 유니티의 Update() 함수가 매 프레임마다 호출될 때, PacketQueue에 쌓인 패킷들을 꺼내서 클라이언트의 PacketHandler로 전달

## 기술 스택

### Server
- Language: C#
- Network: Windows IOCP (TCP)
- Serialization: Protobuf
- Concurrency: Job Queue 기반 구조

### Client
- Engine: Unity
- Language: C#
- Role: 입력 처리 및 서버 상태 반영

## 인상깊었던 학습 내용

**1. "매번 통신 코드를 직접 짜야 할까?" -> Protobuf 패킷 자동화**

- Before) 패킷이 늘어날 때마다 C# 클래스를 일일이 타이핑하고 직렬화 로직을 수동으로 작성함.  
          기능이 추가될수록 작업 시간이 비효율적으로 늘어나고 오타로 인한 버그 위험 존재

- After) 구글 Protobuf를 도입하여 통신 규격을 정의, 패킷 클래스 생성과 핸들러 분기 코드를 자동화하여 작업 시간을 단축 

**2. "여러 명이 동시에 공격하면 왜 서버가 죽을까?" -> JobQueue**

- Before) 유저들의 입장/퇴장, 이동, 스킬 사용 등 공유 자원에 접근할 때마다 lock을 다수 사용  
          규모가 커질수록 락 경합이 심해질 우려가 있고 이는 **데드락(Deadlock)** 위험과 이어질 가능성 높음

- After) 이를 해결하기 위해 작업을 즉시 실행하지 않고 Queue에 담아두는 **JobQueue** 개념을 학습하고 적용  
         여러 스레드가 작업을 큐에 밀어 넣기만 하고, 단일 스레드가 순서대로 꺼내서 처리하게 함으로써 스레드 충돌 방지

**3. "유저가 전투정보를 임의로 조작한다면?" -> 서버 권한 처리**

- Before) 유저가 몬스터에게 피격받아도 체력이 줄어들지않게 하거나, 재화를 임의로 조작  
          -> 무한정으로 만드는 행위 등 개발 의도와는 다르게 플레이 될 위험이 존재

- After) 이를 막기 위해 클라이언트는 스킬을 쓴다는 입력만 서버로 보냄  
         -> 실제 투사체의 이동, 타격 판정, 체력 차감, 몬스터 사망 처리는 모두 서버가 직접 계산하고 결정

**4. "스탯 정보를 단순하게 하드코딩해도 될까?" -> JSON 데이터 파싱**

- Before) 캐릭터 스탯이나 스킬 데미지 수치를 코드에 직접 입력함.  
          -> 이후 밸런스 수정을 할때마다 코드를 다시 컴파일해야 하고 유지보수가 매우 번거로움  

- After) StatData.json, SkillData.json 파일로 데이터를 분리  
         서버/클라이언트 실행 시 이를 파싱하여 메모리에 올려두고 사용하는 DataManager를 구현    
         -> 코드 수정 없이 데이터만 변경하여 신속한 밸런스 조정이 가능하게 함

## 로드맵

1. **브로드캐스팅 병목 해결**
   - 하나의 룸에 있는 모든 유저에게 상태를 브로드캐스트 -> 패킷 복사 오버헤드 부담 증가  
   - 맵을 구역으로 나누는 **그리드(Grid) 기반 시야 처리** 로직을 도입하여 개선 예정  
 
2. **데이터베이스 연동**
   - Entity Framework Core를 활용하여 플레이어 인벤토리 및 저장 시스템 구축

3. **더미 클라이언트(Dummy Client)** 생성 및 부하 테스트
    - 서버가 몇 명까지 버티는지 수치적인 증명
    - C# 콘솔로 이동/공격 패킷 등을 쏘아대는 더미 클라이언트 생성

    -> CPU 점유율과 메모리가 어떻게 변하는지 확인하여 자료 구성
